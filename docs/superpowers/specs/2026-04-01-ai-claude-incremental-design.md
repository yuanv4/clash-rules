# Claude / AI 增量分流设计

**目标**

在不破坏原有订阅配置结构的前提下，为现有 Mihomo / Clash Meta 订阅增量追加 `Claude` 组、`AI` 组，以及对应的分流规则。

**范围**

- 保留原订阅已有的 `proxy-groups`
- 保留原订阅已有的 `rules`
- 保留原订阅已有的 `rule-providers`
- 保留原订阅已有的 DNS、TUN、sniffer 等运行配置
- 仅补充 `claude` / `ai` 规则提供者
- 仅补充 `Claude` / `AI` 两个策略组
- 仅补充 `RULE-SET,claude,...` 和 `RULE-SET,ai,...` 两条规则

**非目标**

- 不再整包覆写完整规则体系
- 不再强制注入通用代理、直连、拦截、兜底等策略组
- 不再修改 fake-ip、nameserver-policy、fallback-filter
- 不再接管订阅的主出站选择逻辑

## 方案对比

### 方案 A：仅追加两个策略组和两条规则

将 `Claude` / `AI` 作为新增组附加到现有配置中，并只补充这两个组所需的规则提供者和规则。

优点：

- 改动最小
- 与原订阅兼容性最好
- 用户原有策略组和规则顺序基本保持不变

缺点：

- 需要在脚本里做一层“寻找默认上游代理组”的兼容逻辑

### 方案 B：只追加规则，不追加组

把 `claude` / `ai` 规则直接指向订阅中现有的主代理组。

优点：

- 实现最简单

缺点：

- 不满足用户需要单独 `Claude` / `AI` 组的目标

### 方案 C：追加组，但完全不做节点筛选

新增组统一引用订阅现有主代理组，不根据地区筛选节点。

优点：

- 比方案 A 更容易实现

缺点：

- 丢失现有脚本对 Claude / AI 节点可用性的优化

**推荐：方案 A**

在保持实现简化的前提下，既满足新增两个独立策略组的目标，也保留最基础的节点筛选能力。

## 架构设计

脚本入口仍保留 `main(config)`，但处理方式从“整包覆盖”改为“增量合并”。

核心拆分为四个动作：

1. 读取原订阅的现有配置
2. 补充 `claude` / `ai` 两个 rule provider
3. 追加 `Claude` / `AI` 两个策略组
4. 把 `RULE-SET,claude,...` 和 `RULE-SET,ai,...` 插入到现有规则前部

## 组件设计

### 1. 默认上游组选择

由于不同订阅的主代理组名称不统一，脚本需要从现有 `proxy-groups` 中推导一个可复用的上游组，作为 `Claude` / `AI` 的兜底出口。

选择顺序：

1. 匹配常见主组名，如 `🚀 节点选择`、`Proxy`、`PROXY`、`节点选择`
2. 若未命中，则选择第一个 `select` / `url-test` / `fallback` 类型的现有组
3. 若仍不存在，则回退到 `DIRECT`

### 2. 策略组追加

新增两个组：

- `🧠 Claude`
- `🤖 AI`

分组策略：

- 组类型保持简化，优先使用 `select`
- 每个组内首先放入匹配筛选条件的候选节点
- 再追加上游主代理组
- 最后追加 `DIRECT`

这样即使某些订阅没有符合地区筛选的节点，组本身仍然可用。

### 3. 节点筛选

保留当前脚本里最有价值的筛选部分，但只用于新增组，不影响订阅原有组。

- `Claude` 倾向日本节点
- `AI` 倾向日本 / 新加坡 / 美国节点
- 继续排除订阅中的说明性伪节点

若筛选结果为空，不报错，直接依赖上游主代理组兜底。

### 4. 规则提供者与规则插入

新增的 `rule-providers` 仅包含：

- `claude`
- `ai`

规则插入顺序：

1. `RULE-SET,claude,🧠 Claude`
2. `RULE-SET,ai,🤖 AI`
3. 原订阅已有规则

这样能确保 Claude / AI 相关域名优先命中新分流，而不破坏原有其他规则。

若原订阅没有 `rules` 数组，则创建数组并写入上述两条规则。

## 错误处理

- 若订阅没有任何节点且没有任何代理组，维持当前的基础报错逻辑
- 若订阅没有 `rule-providers`，初始化为空对象后再注入
- 若订阅没有 `proxy-groups`，初始化为空数组后再追加
- 若订阅没有 `rules`，初始化为空数组后再插入
- 若发现同名组或同名 provider 已存在，则跳过重复注入

## 测试策略

至少覆盖以下场景：

1. 原订阅已有完整 `proxy-groups` / `rules` / `rule-providers`
   预期：仅追加 `Claude` / `AI` 相关配置
2. 原订阅缺少 `rules`
   预期：自动创建并写入两条规则
3. 原订阅缺少 `rule-providers`
   预期：自动创建并写入两个 provider
4. 原订阅无日本 / 新加坡 / 美国节点
   预期：`Claude` / `AI` 组仍存在，并回退到主代理组或 `DIRECT`
5. 原订阅已存在同名 `Claude` / `AI` 组或 provider
   预期：不重复插入
